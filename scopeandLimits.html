There are two fundamental aspects of ToDoTopia that are required to for a minimum viable product (MVP).

  1. Completion of a rudimentary game engine that can show case the basis of:
    1. Game play of the computer game
    2. Cloud based Web application used for hosting the game
  2. API connection between work in progress (WIP) application and ToDoTopia. For a MVP to be completed the following Scope and limits will need to be adhered to.

ToDoTopia is a games platform and will ultimately have the potential to host many games. However, it is important that for the sake of Scope that we limit the development for now, to just one game. Additionally due to time constraints the game should be limited in its functionality until the API and Web platform have been developed to MVP.
is proposed that the game be limited to:

  • 1v1 player
  • One Game Map
  • Limited number of units in the game
    • Infantry unit
    • Tank unit
    • Commando unit
    • Scout unit
  • Units that survive a battle can be used in the next Game however, players will need to expend XP to repair battle damage units before the next battle.

It's temping to develop the game further as this is the “fun” part of what Team Thirteen is doing but we need to be able to show a product that works in a short space of time. Further iterations could greatly develop the game to include more units 2v2 and 3v3 player, new maps and units that survive the battle can gain experience (e.g., More resilient, better weapons etc). However, for the sake of MVP and eliminating scope creep the game play will be limited to the above.

It is proposed that the ToDoTopia cloud-based web platform be limited to:

  • Administration area for mangers
  • User Area
  • Army Builder
  • Users Army (show current units and unit health and cost to repair a battle worn unit to full health)
  • Leader Board (this could be taken out if time is limited)
  • Story Point Conversion (sp to xp)

It is proposed that the API connection be limited to one application, Microsoft’s Azure DevOps, using their Kanban Board. We will need to be able to configure the Jira story cards/tasks to that the story points correlate to the number of Xp points in the game. Looking at Azure Documentation it seems the API JSON will allow story points to come through here

This should allow us to isolate the story pints to convert to Xp for the game. There is a lot of metadata that we could pull though from the API but for the sake of MVP it is prudent to limit development to the story points, however it might be prudent to develop the data base to include all the metadata for future use. At this point displaying the Xp in the Kanban Board world require permissions and extra work which is not necessary as Xp can be viewed in the ToDoTopia Web UI. Additionally, calculation of Xp will need to be done in the TodoTopia backend logic.

Some of the elements of the future scope of the game have already been pointed out with various game upgrades and perhaps rolling out several games’ players can choose from. It is important to point out that the ToDoTopia platform could be developed further to include:

  • employee metrics so that managers can better assess teams and individuals
  • rewards points, connection to frequent flyer points, Woolworths rewards or flybys would further incentivise employees with respect to WIP. Not only could Xp be exchanged for buying units bit there could also be an exchange for rewards points. Additionally, Rewards points could be given as prizes in the game for winners of battle rounds or leader board positions. Further incentivising WIP.

For now, these are out of scope as the focus is limited to getting the platform working. However, with the correct approach Rewards pints could be added provided the relevant API’s are available.
